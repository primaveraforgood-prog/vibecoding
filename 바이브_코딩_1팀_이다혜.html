바이브 코딩
AI와 함께하는 개발 방식의 변화

Andrej Karpathy가 제시한 ‘AI 주도형 개발’ 개념을 중심으로

작성자: 이다혜(경영 20, 1팀) · HTML 발표자료

Vibe Coding Presentation1 / 15
01. Why it matters

코딩의 중심이
“직접 작성”에서 “AI와 대화”로 이동하고 있다

기존 개발은 사람이 코드를 세밀하게 작성하고 오류를 직접 추적하는 방식이었다. 바이브 코딩은 이 과정에서 AI가 더 큰 역할을 맡는다는 점에서 새로운 개발 흐름으로 볼 수 있다.

생성형 AILLM자동 코드 생성개발 생산성
핵심 문제의식2 / 15
02. Definition

바이브 코딩이란?

개발자가 코드의 세부 내용을 모두 직접 통제하기보다, AI에게 요구사항을 말하고 결과를 실행·수정하며 흐름을 타는 방식

Karpathy는 고도화된 LLM을 활용하면 전통적인 코딩 절차의 일부가 대화형 작업으로 대체될 수 있다고 설명했다.

개념 정의3 / 15
03. Karpathy’s idea

핵심은 “AI의 힘을 받아들이는 것”

  • LLM이 단순 코드 작성과 수정 업무를 상당 부분 처리
  • 개발자는 코드의 모든 세부 사항을 기억하지 않아도 됨
  • 사람은 방향을 말하고, AI는 구현과 반복 수정을 담당
Karpathy의 관점4 / 15

작업 방식5 / 15
05. Example

예: “사이드바 여백을 조금 줄여줘”

전통적 방식

관련 CSS 파일을 찾고, 클래스명을 확인하고, 수치를 직접 조정한 뒤 테스트한다.

바이브 코딩

AI에게 수정 의도를 말하고, 제안된 변경사항을 적용한 뒤 결과만 확인한다.

Karpathy는 이런 단순 조정에서 키보드를 거의 의식하지 않는 수준의 작업 방식을 설명했다.

06. Tools

도구의 발전이 바이브 코딩을 가능하게 했다

LLM

AI IDE

반복 실행

기사에서는 Cursor Composer와 Sonnet 같은 도구가 언급된다.

기술적 배경7 / 15
07. Strength

가장 큰 장점은 속도와 진입장벽 완화

  • 아이디어를 빠르게 시제품으로 전환 가능
  • 단순 기능 구현과 UI 수정에 효과적
  • 초보자도 결과물을 만들며 학습할 수 있음
장점8 / 15
08. Limitation

하지만 모든 프로젝트에 적합한 방식은 아니다

오류 해결 한계

AI가 일부 버그를 끝까지 해결하지 못할 수 있다.

코드 이해 부족

개발자가 내부 구조를 모르면 유지보수가 어려워진다.

복잡한 프로젝트

장기·대형 시스템에서는 무작정 적용하기 어렵다.

품질 검증

작동하는 것처럼 보여도 보안·성능 문제가 남을 수 있다.

적합한 활용처10 / 15
10. Risk control

바이브 코딩을 안전하게 쓰는 방법

  • AI가 만든 코드를 그대로 믿지 말고 실행 결과를 확인
  • 중요 기능은 직접 구조를 이해하고 검토
  • 오류 메시지와 수정 내용을 기록해 재현 가능하게 관리
  • 보안·개인정보·결제 기능에는 더 엄격한 검증 필요
활용 원칙11 / 15
11. Public reaction

사람들의 반응은 엇갈렸다

긍정적 반응

코딩 능력의 기준이 바뀔 수 있다는 기대

비판적 반응

12. Comparison

전통적 코딩 vs 바이브 코딩

전통적 코딩

문법과 구조를 사람이 직접 설계
정확성과 통제력이 강점
속도는 상대적으로 느릴 수 있음

바이브 코딩

자연어 지시와 AI 생성에 의존
속도와 접근성이 강점
검증과 유지보수가 핵심 과제
방식 비교13 / 15
13. Implication

우리에게 주는 시사점

앞으로 중요한 능력은 단순히 코드를 외우는 것이 아니라, AI에게 정확히 요구하고 결과를 판단하는 능력이다.

  • 질문을 잘 만드는 능력
  • 결과물을 검토하는 논리력
  • 문제를 작은 단위로 나누는 설계력
학습 관점14 / 15
14. Closing

결론: 바이브 코딩은
개발자를 대체하는 방식이 아니라 개발 방식을 바꾸는 흐름

작고 빠른 작업에서는 강력하지만, 복잡하고 중요한 프로젝트에서는 사람의 이해와 검증이 반드시 필요하다.
